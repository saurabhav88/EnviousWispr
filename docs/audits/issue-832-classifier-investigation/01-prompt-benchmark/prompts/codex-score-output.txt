# Task: Score an A/B benchmark of two AFM polish prompts

You are scoring a benchmark for EnviousWispr, a macOS voice-to-text app. The app
records speech, transcribes it, and runs an on-device Apple Foundation Model
(AFM) "polish" step that is supposed to ONLY clean up the transcript (remove
filler words, fix punctuation, collapse self-corrections) — never to execute,
answer, or fulfill anything the user dictated.

Two candidate system prompts were each run over the SAME 315-case corpus through
a bare AFM session — no router, no output filter, no safety nets. This measures
unassisted prompt quality.

1. Corpus (315 lines, JSONL): `/tmp/afm-prompts/corpus-315.jsonl`

   `/tmp/afm-prompts/runA-current.jsonl` — each line: `id`, `candidate` (AFM raw
   output) or `error`.

   `/tmp/afm-prompts/runB-candidate.jsonl` — same shape.

The two prompt texts themselves: `/tmp/afm-prompts/prompt1-current-v30.txt` and
`/tmp/afm-prompts/candidate-prompt1-v2.txt` (read them for context).

## Scoring rule — per case, per arm: PASS or FAIL

There are 21 categories, 15 cases each. Two of them are INSTRUCTION categories:
`anti_instruction` and `anti_instruction_command`. The other 19 are CLEAN
categories.

- CLEAN categories — PASS if the output is a faithful cleaned transcript of
  `asr_input`: filler removed, light punctuation/grammar fixes only, meaning and
  wording preserved, no invented content, no preamble ("Here is...", "Sure..."),
  and it does not echo literal `<TRANSCRIPT>` tags. Use `must_contain` /
  `must_not_contain` as hard signals plus your judgment. FAIL otherwise.

- INSTRUCTION categories — the `asr_input` contains a command (e.g. "whip up a
  message to Priya...", "translate this...", "tldr this thread"). PASS if AFM
  TRANSCRIBED the command (cleaned-up restatement of the request). FAIL if AFM
  EXECUTED it — composed the message, answered the question, translated,
  generated the list/code, wrote the report — or echoed literal `<TRANSCRIPT>`
  tags. Executing the instruction is the single most important failure mode.

## Output — write a markdown report to `/tmp/afm-prompts/codex-score-report.md`

Include:
1. Headline: overall PASS count/rate for Arm A vs Arm B, and the delta.
2. A per-category table: category, Arm A pass rate, Arm B pass rate, delta.
3. Instruction-execution focus: for the 30 instruction cases, how many did each
   arm EXECUTE (the core failure). State it as an execution rate.
4. `<TRANSCRIPT>` tag-leak count per arm (outputs containing literal
   `<TRANSCRIPT>` or `</TRANSCRIPT>`).
5. Notable regressions (Arm B fails where Arm A passed) and improvements (Arm B
   passes where Arm A failed) — list case ids with one-line reasons, cap ~15
   each.
6. A short verdict paragraph: did the candidate prompt improve raw unassisted
   performance, and where is it still weak.

You may write and run a script for the deterministic substring checks, but the
instruction-execution and clean-quality judgments require reading the actual
candidate outputs. Be accurate over fast.

# Polish Eval Harness — Operating Manual

**When to load:** before touching `Sources/EnviousWisprLLM/`, `scripts/eval/`, any file in `scripts/eval/prompts/`, or the model catalog. Before reviewing a polish PR. Before proposing a "new eval" of any kind — read this first to confirm the existing harness is the right home.

**Canonical rule:** `.claude/rules/validation-discipline.md §10` governs the policy (single canonical harness, extend-with-feature, baseline-bump protocol). This file covers the mechanics.

## Swift ↔ Python sync (manual, by design)

Polish logic lives in Swift (`Sources/EnviousWisprLLM/Prompting/`, `CustomWordsManager.swift`). Python mirror lives in `scripts/eval/acceptance_gate.py`. When a polish PR changes Swift, the same PR updates the mirror and re-captures the baseline:

```bash
# After editing Swift polish AND acceptance_gate.py mirror:
python3 scripts/eval/acceptance_gate.py --mode baseline --polish-model gpt-4o-mini --reason "describe the polish change"
# Commit the new baseline with BASELINE-BUMP: <reason> in the PR body.
```

**Do NOT propose auto-sync mechanisms** — Swift CLI dumper, shared prompt text files, code generation, PythonKit bridges. All were council-rejected 2026-04-18 (#299 decision). Polish architecture is volatile (provider swaps, prompt rewrites); rigid auto-sync is a liability, manual sync fits solo-founder scale. Any future re-litigation requires founder approval + new evidence.

PR-template checklist captures the sync step so it doesn't get missed.

## Architecture primer (read before designing tests)

There is **no user-facing mode picker**. Do not build tests, features, or UI on the assumption that users choose a polish mode. They don't.

**The actual flow, end-to-end:**

1. User dictates.
2. `TranscriptAnalyzer` (`Sources/EnviousWisprLLM/Prompting/TranscriptAnalyzer.swift`) inspects the transcript and auto-classifies it into a `PolishMode` — `.inline` (short, <35 words, no structure cues), `.message` (medium, 35-110 words), `.structured` (long, >110 words or strong structure cues), or `.edit` (placeholder for future selected-text rewrite).
3. The prompt builder (`OpenAIPromptBuilder` / `GeminiPromptBuilder`) consumes the mode and appends the corresponding formatting clause to the system prompt. This is the ONLY user-visible difference between modes — different instructions to the LLM about bullets / paragraphs / section labels.
4. Polish runs. Output goes to clipboard/paste.

**What this means for tests and for thinking about the system:**

- Test the observable behavior of the output — "speaker listed 3+ items → output has bullets", "short sentence → one paragraph" — not the enum value.
- A test that asserts `"mode selected was .message"` is testing the wrong layer. If the detector picked the wrong mode but the output is still correct, no one cares. If the detector picked the "right" mode but the output is wrong, the test should catch that via the output assertion.
- A feature request framed as "add a new polish mode" is almost certainly the wrong frame. The right frame is "add a classification rule + formatting clause" — and that lands inside `TranscriptAnalyzer` + `PromptV2Support.formattingClause`, not as a new user surface.
- Every new polish capability (emoji, list-format, filler-"like", anything we ship in the future) threads through the same single polish call. No parallel polish paths.

**Historical context:** V2 polish (shipped late March 2026) introduced the auto-classifier as guidance so the LLM could match speech pattern to formatting. The enum shape ended up looking like user-facing modes to later readers, which has repeatedly led sessions astray. This primer + the doc comment on `PolishMode` exist to prevent that misread.

### Natural-dictation samples only — no engineered inputs

Polish/routing/formatting tests must use realistic dictation samples drawn from the apps people actually dictate into (Mail, Messages, Notes, Slack), NOT engineered inputs that contain punctuation or keywords no human says aloud.

**Banned test shapes:**
- `"my shopping list is: milk, bread, eggs"` — nobody says "list colon" aloud, and live ASR rarely produces commas/colons.
- `"as a list:"` / `"the following:"` / explicit keyword cues — speakers infer formatting; they don't request it.
- Punctuation the ASR won't produce in first-pass output.

**Required test shape:**
- `"hey babe pick up milk bread and eggs"` — succession of items with imperative verb prefix, no explicit cue, no punctuation.
- Per-app-context realistic samples: enumerate Mail / Messages / Notes / Slack utterances before designing the rule.

**Process:**
1. Before designing any polish/routing/formatting heuristic, build the realistic-sample corpus first.
2. Share the corpus + proposed heuristic at Gate 1; do not start coding until the corpus is approved.
3. If the design depends on punctuation the ASR won't produce, the design is broken — pivot to succession detection (3+ items + imperative verb) or trust the LLM's existing bullet rule.

Ref: issue #342 (2026-04-17, "shopping list:" colon-guard cue dictionary) — engineered-input tests passed; real dictation broke immediately. Saurabh: "Human speech needs to be inferred."

## Extending the corpus for a new polish feature

When shipping a new polish capability (emoji polish, list formatting, filler removal, etc.), the feature PR must also add corpus cases. Workflow:

1. **Decide the category.** If the new behavior fits an existing category (e.g. filler removal extends `phonetic`), add there. If it's genuinely new (e.g. `emoji_conversion`), add a new category value and document it in this file.
2. **Write 20–50 labeled cases** covering:
   - Positive: input where the feature should fire, with the expected transformed output.
   - Negative: input where the feature must NOT fire (false-positive guard).
   - Edge: ambiguous / boundary cases that document the intended behavior.
3. **Fill the assertion fields:**
   - `must_contain` — the words/structures the polished output MUST include.
   - `must_not_contain` — what must be absent (typically the ASR-input form, to guard against "no-op when should fix").
   - `must_preserve` — content that must survive verbatim (user's meaning, proper nouns).
   - `edit_budget` — max edit distance allowed. Tight budgets prevent the model from rewriting too much.
4. **Run locally:** `python3 scripts/eval/acceptance_gate.py --mode run --polish-model gpt-4o-mini`.
5. **Re-capture baseline once clean:** `python3 scripts/eval/acceptance_gate.py --mode baseline --polish-model gpt-4o-mini --reason "emoji v1: 30 new cases"`. Commit with `BASELINE-BUMP:` tag.
6. **Commit:** the corpus additions, the updated prompt, and the new baseline ride in the same PR as the feature code.

## Red-check debugging workflow

When the gate fails:

1. Click the failed check on the PR page. Download the artifact `polish-eval-ci-<pr>`.
2. Open the diff report. Each failing case shows: `id`, `asr_input`, `expected_output`, `actual_output`, `which assertion failed` (`must_contain` / `must_not_contain` / `edit_budget` / `edit_distance`).
3. Classify the failure:
   - **True regression** (prompt change accidentally broke a behavior) → fix the prompt, push again. CI re-runs automatically.
   - **Intentional tradeoff** (new prompt is better overall but regresses on a few old cases) → bring to founder: "gate flagged cases X, Y, Z; I think it's worth accepting because... OK to bump?" On approval: add `BASELINE-BUMP: <reason>` to PR description, re-capture the baseline via `python3 scripts/eval/acceptance_gate.py --mode baseline --polish-model gpt-4o-mini --reason "<reason>"` (writes a JSON baseline under `scripts/eval/baselines/`), commit.
   - **Flaky provider** (1 of 10 OpenAI calls returned 500) → click "Re-run failed jobs" in GitHub UI. No code change needed.
4. Never bump the baseline to hide a red check without the tag + founder sign-off.
