The Developer Workflow

Click any step to learn more, or press Play to watch the full journey of your code from idea to main.

Think of main like a published book. You never scribble edits directly into the published copy — you draft changes on separate scratch paper first, then submit them for a quick quality check before they make it into the next edition.
Key: 🧑 YOU Steps where you need to take action 🤖 AUTO Steps that happen automatically — no action needed

01 Create Your Scratch Copy 🧑 YOU

Make a separate copy of the code just for your new idea. The published version (main) stays untouched.

Step 1: Run this in your terminal — replace the name with your feature:

$ git checkout -b feat/add-dark-mode
  • Name it after what you're building: feat/ for features, fix/ for bug fixes, docs/ for docs
  • You now have your own copy of the code to experiment in — nothing you do here touches main
  • Think of it as a sandbox: break things, try ideas, nothing is permanent yet

02 Write Your Code 🧑 YOU

Make changes, add features, fix bugs. Work at your own pace — nothing here is permanent until you say so.

  • Step 2: Write code and save your progress with git commit -m "what I did" as often as you like
  • You can save progress in small chunks — ten tiny commits is fine, nobody will see them
  • Nothing you do here changes main until the very end of this process
  • Made a mistake? No problem — your scratch copy is isolated

03 Upload Your Changes to GitHub 🧑 YOU

Send your scratch copy up to GitHub so it's backed up and ready to review. Your copy only — main is still untouched.

Step 3: Run these three commands:

$ git add -A
$ git commit -m "feat: add dark mode toggle"
$ git push -u origin feat/add-dark-mode
  • You can push to your own copy anytime — no restrictions, no approval needed
  • The -u flag is only needed the first time you push a new copy (it links your computer to GitHub)
  • GitHub automatically scans for accidentally pasted API keys or passwords as a safety net

04 Submit Your Changes for Review 🧑 YOU

On GitHub, click "Open Pull Request" to say: "I'm ready — please check this over before it goes live."

Step 4: On GitHub, click the green "Compare & pull request" button that appears after your push. Then these things happen automatically:

  • 🤖 AUTO A pre-filled checklist appears — what changed, what to verify, any release notes needed
  • 🤖 AUTO You are assigned as the reviewer (since you own the whole project)
  • 🤖 AUTO The quality checks start running immediately in the background
  • The green "Merge" button stays locked until everything passes

Files behind this:

.github/pull_request_template.md .github/CODEOWNERS

05 Automated Quality Checks 🤖 AUTO

GitHub automatically builds your code three different ways to catch problems before they go live. You don't do anything — just wait.

Three checks run automatically on a fresh Mac in the cloud:

✓ Normal Build — does it compile at all?
✓ Release Build — does it compile the way users would get it?
✓ Test Build — do the automated tests still compile?
  • Cached packages keep it fast: usually done in about 1 minute 40 seconds
  • If any check fails, the Merge button stays locked — nothing broken can reach main
  • Watch them run live in the "Checks" tab on the PR page

File: .github/workflows/pr-check.yml

06 Read Your Own Changes 🧑 YOU

Before merging, take a few minutes to read what you actually changed. This single habit catches an astonishing number of bugs.

Step 6: On the PR page, read the "Files changed" tab — everything highlighted in green is new, red is removed.

  • Seeing your changes laid out side-by-side catches things you overlooked while writing
  • Work through the pre-filled checklist: tests, docs, anything to mention in release notes?
  • Since you own the whole project, you approve your own PR — click "Approve" in the review panel
  • You can also paste the diff into Claude and ask "does anything look off here?"
  • Any open comments or questions on the PR must be marked resolved before merging

07 Merge — Ship It 🧑 YOU

When everything passes, click Merge. Your changes are now live in main, squashed into one clean entry in the history.

Step 7: The green "Merge" button becomes clickable only when:

  • All three automated checks passed
  • You approved the review
  • All open comments are resolved

What "squash" means: Even if you made 47 messy "fix typo" commits on your scratch copy, main gets just one clean, tidy entry that says what the feature was. It's like turning a rough draft into a final published paragraph — all the messy in-progress saves disappear. This is enforced automatically; you can't accidentally do it the messy way.

08 Tidy Up 🧑 YOU

Delete the scratch copy — it's done its job. Keep things tidy for next time.

Step 8: GitHub shows a "Delete branch" button right after you merge — click it. Then on your computer:

$ git checkout main
$ git pull
$ git branch -d feat/add-dark-mode
  • Don't worry — your changes are already safely in main. Deleting the scratch copy doesn't lose anything
  • You're now back on the latest version of main with your feature included
  • Ready to start the whole cycle again for the next thing
⚠

What Happens If CI Fails?

▼

When any automated check fails, the Merge button stays locked. Nothing broken can reach main. Here is what it looks like:

✓ Normal Build — passed in 47s
✗ Release Build — failed in 1m 12s
◯ Test Build — skipped (waiting on Release Build)

🧑 YOU What to do:

  1. Click the red "failed" check on the PR page — it opens the full error log
  2. Find the error message (usually near the bottom of the log)
  3. Fix it in your code on your scratch copy
  4. Save and push again — 🤖 AUTO the checks start over automatically
  5. Once all three turn green, the Merge button unlocks
✓

This is the whole point. You catch build problems before they reach users. Without this, a broken build could ship to users for days before anyone notices.

What Was Set Up

Six layers of protection, each one solving a specific problem that trips up solo developers.

Think of these as safety rails on a highway. You still drive the car — but the rails are there if you drift. Most of the time you won't notice them. When you do, they'll have saved you from something bad.
Key: 🧑 YOU Needs a one-time setup action from you 🤖 AUTO Runs on its own — nothing for you to do

Main Branch Lock 🤖 AUTO

main is locked — nobody can push directly to it, not even you. Every change must go through the full PR process. There is no way to accidentally skip the checks.

ENFORCED

Automated Build Checks 🤖 AUTO

Every time you open a PR, GitHub automatically builds your code three ways in the cloud. You don't run these — they just run. Usually done in about 1 minute 40 seconds.

AUTOMATIC

Pre-filled PR Checklist 🤖 AUTO

When you open a PR, a checklist automatically appears: what changed, what to test, whether release notes are needed. It's like a reminder card so you never forget a step.

AUTOMATIC

Auto-Assigned Reviewer 🤖 AUTO

Every PR automatically tags you (@saurabhav88) as a reviewer via CODEOWNERS. Merging requires only the build-check CI to pass — no human approval is enforced by branch protection.

AUTOMATIC

Weekly Dependency Updates 🤖 AUTO

Every Monday, GitHub automatically checks if any of your code packages have newer versions. If they do, it opens a PR with the update ready to review — like a personal assistant that tracks software updates for you.

WEEKLY

Secret & Vulnerability Guard 🤖 AUTO

Three things happen automatically: known security bugs in your packages trigger alerts; if you accidentally paste an API key into your code, the push gets blocked; any leaked tokens already in the repo get flagged.

ALWAYS ON

Why This Matters

Every piece of this solves a real problem that solo developers hit eventually. Usually at the worst possible moment.

Solo developers skip these guardrails because it feels like extra work. It's not — it's insurance. The overhead is maybe 5 extra minutes per change. The time saved catching one bad deployment before users see it is measured in hours.

You Can't Accidentally Ship a Broken Build

Without this, one mistyped command (git push straight to main) can send broken code to every user. Now that literally cannot happen — every change is locked behind the automated checks.

Reading Your Own Code Before Shipping It Catches Bugs

When you're in the middle of writing code, you miss things. Stepping back and reading your changes as a diff — highlighted red and green in the PR — catches an astonishing number of mistakes. The PR process forces a natural pause before shipping.

"Works on My Machine" Is Not Good Enough

Code that compiles fine on your Mac might fail in a clean environment — different cached files, different tool versions, different defaults. The automated checks run on a fresh Mac in the cloud every time, catching those invisible differences before users see them.

Your Code History Stays Readable

All those in-progress saves ("fix typo", "actually fix it", "ok REALLY fixed now") get collapsed into one clean commit when you merge. Six months from now when something breaks, you can scan the history and immediately see: what changed, when, and what it was for — not 47 lines of noise.

Accidentally Pasting a Password Into Code Won't Ruin You

It happens to experienced developers all the time — you copy an API key to test something and forget to remove it before committing. Push protection catches it before it reaches GitHub and blocks the upload entirely. Without this, that key is now public and you have minutes to revoke it before bots find it.
