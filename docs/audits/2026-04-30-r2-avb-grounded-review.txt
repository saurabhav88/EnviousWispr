**Premise Check**

1. The real cost is not just `public`; it is that `EnviousWisprPipeline` still imports WhisperKit directly and stores/constructs WhisperKit-specific worker types (`WhisperKitPipeline.swift:9`, `:154`, `:1121-1129`). Narrowing access without removing that compile-time dependency only partially fixes the boundary.

2. Heart-path safety is overclaimed for any redesign unless it preserves today’s fallback shape: LID abstain falls back to WhisperKit’s internal language detection (`WhisperKitPipeline.swift:675-701`), the incremental worker is only enabled in locked-language mode (`WhisperKitPipeline.swift:467-475`), and failed/stale worker output falls back to batch transcription (`WhisperKitPipeline.swift:765-820`).

**Opinion**

Between A and B, pick **A** for R2. Over 2-3 years, B is cleaner in principle but too easy to overmove. The pipeline currently owns session state, audio capture branching, cancellation, fallback, telemetry, empty-result handling, finalization, paste context, and model-unload policy (`WhisperKitPipeline.swift:397-502`, `:523-820`, `:824-840`). `EnviousWisprASR` is not just a thin wrapper: it already owns backend selection/lifecycle (`ASRManager.swift:5-8`, `:61-105`), backend APIs (`ASRProtocol.swift:14-47`), WhisperKit batch decode (`WhisperKitBackend.swift:129-145`), Parakeet streaming (`ParakeetBackend.swift:107-165`), LID (`LanguageDetector.swift:60-63`, `:116-121`), and the incremental worker (`WhisperKitIncrementalWorker.swift:23-48`). But it does not own the dictation session. Moving “run an incremental decoding session for me with callback” into ASR risks pulling capture-provider semantics and fallback policy down with it.

A contains upstream WhisperKit churn better than today because `detectLangauge`, `transcribe(audioArray:decodeOptions:)`, `DecodingOptions`, and tokenizer construction stay in ASR-owned code (`LanguageDetector.swift:567-571`, `WhisperKitIncrementalWorker.swift:135-145`, `:217-242`, `WhisperKitBackend.swift:158-190`). If a third backend arrives, A does not pressure Parakeet and WhisperKit into one abstraction; that matters because the repo explicitly already has a generic `ASRBackend` with optional streaming hooks (`ASRProtocol.swift:32-47`) and Parakeet’s behavior is structurally different (`ParakeetBackend.swift:24`, `:107-165`). Debugging is also simpler: pipeline bugs stay in the pipeline, engine bugs stay in ASR.

The unconsidered stronger **C** is: create an opaque WhisperKit runtime/session interface, not a broad ASRBackend expansion. Concretely, delete the tokenizer parameter, make `WhisperKitIncrementalWorker` conform to a small `WhisperKitIncrementalSession` protocol with `start`, `finalize`, and `cancel` matching its current public methods (`WhisperKitIncrementalWorker.swift:48-69`, `:196-200`), and have `WhisperKitBackend` vend `makeIncrementalSession(options:) -> (any WhisperKitIncrementalSession)?`. Then `WhisperKitPipeline` stores `any WhisperKitIncrementalSession` instead of `WhisperKitIncrementalWorker` (`WhisperKitPipeline.swift:154`, `:765-768`, `:1089-1090`). That removes the pipeline’s need to import WhisperKit (`WhisperKitPipeline.swift:9`) and avoids a generic ASR mega-protocol.

For LID, the genuinely correct long-term fix is not “bridge returns a WhisperKit handle.” It is to stop passing `WhisperKit` into `LanguageDetector.detect` (`LanguageDetector.swift:116-121`). The clean shape is: `WhisperKitBackend` performs the raw WhisperKit window calls internally and returns Sendable language observations; `LanguageDetector` keeps only memory/classification. Today the non-Sendable hop is real: pipeline marks the handle `nonisolated(unsafe)` (`WhisperKitPipeline.swift:681-684`), then the detector actor calls `whisperKit.detectLangauge` (`LanguageDetector.swift:520-524`, `:565-571`). R2 should not attempt that full split unless it is willing to touch LID internals; otherwise say honestly that access narrowing does not fix isolation.

Tokenizer is dead code. It is stored at `WhisperKitIncrementalWorker.swift:26`, initialized at `:40-45`, and never read in the worker’s decode paths (`:128-145`, `:217-242`). The pipeline fetch at `WhisperKitPipeline.swift:1127` can be deleted rather than wrapped.

Testing: A only gives a useful unit seam if the bridge returns a mockable worker protocol. A bridge that returns concrete `WhisperKitIncrementalWorker` still needs real `WhisperKit` state (`WhisperKitIncrementalWorker.swift:24-26`, `:40-45`). B only improves testing if ASR exposes a protocol seam; a concrete ASR helper is integration-only. The existing tests already show this distinction: `FakeASRBackend` works because `ASRBackend` is a protocol (`ASRManagerBackendInjectionTests.swift:105-120`), while WhisperKit pipeline-level tests avoid constructing a real backend because that dependency is heavy (`HeartPathTelemetryWiringTests.swift:34-40`).

Doing nothing probably breaks nothing for users for a long time. The leaks are architectural, not current heart-path defects: batch fallback exists (`WhisperKitPipeline.swift:800-820`), LID abstain preserves transcription (`WhisperKitPipeline.swift:697-701`), and missing worker just means batch path (`WhisperKitPipeline.swift:815-820`). The concrete cost is future drift: more pipeline code can grab `whisperKitInstance` or tokenizer because they remain public (`WhisperKitBackend.swift:33-36`), and upstream API churn can keep leaking across the module boundary.

**Sign-off: mixed/C.** For R2, implement A-sized changes, but make the seam C-shaped: delete tokenizer, return an opaque incremental-session protocol, and route LID through a narrow ASR method only if you accept that it still preserves the current unsafe hop. What would change my mind: if the team commits to splitting `LanguageDetector` into “WhisperKit raw observation” plus “Sendable classifier/memory” now, then a fuller B-style relocation becomes worth it because it would actually remove the non-Sendable boundary instead of hiding it.
