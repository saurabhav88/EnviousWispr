Background

What IS the Brain?

The "brain" is a collection of files that tell Claude — the AI that builds this app — everything it needs to know about the project. Think of it as an instruction manual that Claude reads at the start of every work session. Without it, Claude would have to re-explore the codebase from scratch each time. With it, Claude walks in like a senior employee who already knows the system.

1
The Playbook — CLAUDE.md
The master instruction file. Like a team lead's handbook: "here's how we work, here are the rules, here's who does what." Every session starts by reading this. It also lists every specialist and when to use them.

Architecture docs, coding conventions, known gotchas, feature catalog, file indexes, roadmap. Claude reads these to understand the codebase without re-exploring everything from scratch. Like a Wikipedia that's specific to this project.

Virtual team members, each an expert in one domain: audio-pipeline, build-compile, testing, wispr-eyes (UI verification), frontend-designer, and more. Instead of Claude doing everything itself, it delegates to the right specialist — like a manager routing work.

Before the Surgery

The Problem We Found

An audit in early March revealed that the brain's structure was excellent — but it hadn't been keeping up with reality. The skeleton was solid. The information inside was starting to go stale.

The Numbers Were Wrong
The brain said there were 26 SettingKeys — actually 34. Said 68 Swift files — actually 72. Said 8 features done — actually 16+. Imagine handing a new employee an org chart that's six months out of date. They'd make decisions based on wrong information.
Real example: SettingKey count appeared in 4 different files with 4 different answers
feature-catalog.md 26
type-index.md 27
file-index.md 31
Source code (reality) 34
No Self-Maintenance
Zero mechanisms to keep the brain current. No scripts, no hooks, no reminders. Every time a feature shipped, the brain got a little more stale. Like a wiki that nobody updates — useful at first, gradually misleading.
Isolated Islands
48 plan files in docs/plans/ — completely disconnected from the brain. 8 knowledge files that no agent ever reads. Information existed, but wasn't wired up to the people who needed it.
The Good News (what was already working)
Zero broken links — every reference pointed to a real file. Zero orphan files. 88 total brain files, all reachable. The structure was excellent. It just wasn't maintaining itself.
The Fixes

What We Did About It

Four targeted interventions — each addressing a specific failure mode. No structural overhaul needed; the architecture was already good.

📋
Fix 1: Created the Brain Manifest
brain-manifest.md — new file

Every brain file now has a classification: Generated (auto-regenerate from code), Canonical (human-authored truth), Reference (stable docs), or Transient (temporary plans). When two files disagree on a number, the manifest says which source is the authority.

Why it matters: No more "wait, which file has the right number?" confusion. The manifest is the constitution — it defines who owns what, and who wins when there's a conflict.
🔄
Fix 2: Built Auto-Refresh Scripts
brain-refresh.sh + brain-check.sh — new scripts

brain-refresh.sh scans the actual source code and regenerates 4 index files (file-index, type-index, task-router, feature-catalog) from reality — not from memory. brain-check.sh verifies freshness, like a spell-checker but for staleness. Run one command. Know everything is accurate.

Why it matters: Instead of manually counting files and hoping the docs match, a developer can run one command and know whether the brain is current. "PASSED" or "3 files stale — run brain-refresh."
✅
Fix 3: Corrected All Factual Errors
7 files updated with correct data

Corrected SettingKey counts (26 → 34), file counts (68 → 72), removed 8 duplicate entries, fixed a wrong CI claim (build-compile agent said CI was "required" — it's informational), fixed wrong token names (APPCAST_BOT_TOKEN → GITHUB_TOKEN).

Why it matters: Claude was making decisions based on wrong information. Stale counts create confusion — Claude might skip files it thinks don't exist, or spend time searching for tokens that have been renamed.
✂️
Fix 4: Separated Machine Content from Human Content
Delimiter added to all 4 generated index files

Generated files now have a clear delimiter: everything above the line is auto-generated (the script owns it, don't touch). Everything below is human-authored content, preserved across regeneration. Like a spreadsheet where formulas update automatically but your hand-written notes stay put.

Why it matters: We can refresh indexes without losing the hand-written task routing patterns, feature descriptions, and contextual notes that took time to write.

The Payoff

Why This Matters

None of this is visible in the app itself — it's infrastructure for the AI doing the building. But it directly affects how fast and reliably the app gets built.

⚡
Faster sessions
Claude doesn't waste time re-discovering things that changed since last session, or cross-referencing conflicting numbers to figure out which is correct. Accurate context means faster decisions.
🎯
Fewer mistakes
No more decisions based on stale numbers. If the brain says 34 SettingKeys, there are 34 SettingKeys — because the script just counted them from the source code.
🧠
Self-aware system
The brain now knows what kind of file each piece is and how to keep it fresh. Generated files know they're generated. Canonical files know they're the source of truth.
📈
Scales with the codebase
As the codebase grows, the indexes grow with it automatically. No manual updating of file counts or feature lists every time something ships.

The brain is 88 files working together — 1 playbook, 16 knowledge docs, 11 agents, 46 skills, 1 rules file, and a persistent memory layer. After the surgery, it maintains itself.
