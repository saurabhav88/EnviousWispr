**Sign-Off: PROCEED-WITH-REVISIONS**

The core AppState ceiling idea is sound, but the plan is not baseline-clean as written. The AppState numbers are correct; the cross-module TODO guard and dependency-direction script are the parts that need revision before implementation.

AppState collaborator count: confirmed 19 under the Phase E definition: top-level `let`, non-primitive, including existentials. The list is: `permissions` [AppState.swift:18], `audioCapture` :19, `asrManager` :20, `keychainManager` :21, `hotkeyService` :22, `benchmark` :23, `recordingOverlay` :24, `setup` :29, `audioDeviceList` :30, `captureTelemetry` :31, `pipeline` :38, `whisperKitPipeline` :39, `polishService` :86, `settingsSync` :89, `transcriptCoordinator` :95, `customWordsCoordinator` :111, `customWordsPropagator` :118, `llmDiscovery` :121, `aiAvailability` :124. I agree with counting the two existentials because they are still AppState-held architectural collaborators, even if implementation selection happens in `init`.

Line count: confirmed `wc -l` is 954, so 1050 is a reasonable +10% clean ceiling.

Stored-property parsing: the brace-depth approach works on current `AppState.swift`. Top-level declarations sit at depth 0; method locals and closure locals sit at depth 1+. Lines with both opens/closes, like `guard let self else { return }`, stay inside method/closure depth and are not counted. The primitive exclusions do not miscategorize current top-level slots. `settings = SettingsManager()` is missed because it is `var`, matching the plan’s stated non-goal. `postCompletionWarningTask: Task<Void, Never>?` is excluded, correctly, and is also `var`.

Mirror: do not use it. `AppState.init()` constructs real capture/ASR selections from defaults [AppState.swift:133-148], creates real storage/pipelines/backend objects :158-202, wires `SetupCoordinator` :213-218, applies settings :220-230, schedules Tasks :238 and :409, and wires many live callbacks. It may not always crash or hang, but it absolutely pulls runtime infrastructure into a test that only needs source structure.

Cross-module public-TODO guard: plan claim of zero offenders is wrong. Current proposed broad window flags real and false positives. Real offender: `WhisperKitBackend.makeDecodeOptions` has `TODO: Phase 2 — narrow... cross-module` immediately above a `public func` [WhisperKitBackend.swift:156-157]. False positives include `DictationActivityProviding` due “Narrow contract” [Transcript.swift:89-91], `SilenceDetector` due “three-phase” [SilenceDetector.swift:72-74], `AudioCaptureManager.startCapture` due “single-phase” [AudioCaptureManager.swift:226-227], `TelemetryService.trackLIDAbstained` due `narrow_margin` [TelemetryService.swift:418-419], and `ProgressCallback` due `phaseString` [ASRProtocol.swift:5-8]. Tighten the regex to actual confessional comments, probably requiring `TODO` plus one of `narrow|temporary|cross-module|phase`, rather than raw words in a ±3 window.

Dependency graph: revise before shipping. `Package.swift` says `EnviousWisprASR` depends on Core and Audio [Package.swift:57-65], `EnviousWisprServices` depends only on Core plus vendors :48-55, and `EnviousWisprLLM` depends only on Core :67-70. The proposed graph omits ASR→Audio and adds unused/stale Services→Audio/ASR and LLM→PostProcessing. More importantly, the script loops over all `Sources/*/` except `EnviousWispr`; it will include executable service targets with no `allowed` entries. Baseline would fail on six imports from `EnviousWisprASRService` and `EnviousWisprAudioService` [Package.swift:86-105]. Add allowed entries for those service targets or explicitly skip executable service dirs.

Test placement is fine. `SetupCoordinatorTests.swift` uses `@testable import EnviousWispr` [SetupCoordinatorTests.swift:6] because it constructs internal app-shell types. The new source-reading tests do not need any EnviousWispr import.

Phase F reconciliation: the shipped diff removed two `let`s, added one `let setup`, and removed one `Task` var. Under Phase E’s current metric, post-F is 19. The Phase F plan’s “19 → 17” claim [issue-501 plan:17] used a different/looser “concrete property” concept and its ship-criteria grep expected 17 [issue-501 plan:263], but the live tree does not match that. Document the metric change explicitly so future reviewers do not think Phase F under-delivered.

Completeness: I would not add TranscriptionPipeline or WhisperKitPipeline ceilings in Phase E. Architecture rules already warn pipeline can become the next junk drawer [architecture-rules.md:48-52], and the plan’s §10 follow-up is the right place once pipeline shape stabilizes.

Council-skip is acceptable only after the above fixes.
