01 — WHY WE RESET

Three hours of patches. Five new bugs. One clean answer.

The original WhisperKit integration tried to bolt batch transcription into Parakeet's streaming-first pipeline. Every patch revealed two more edge cases. After consulting GPT-4o and Gemini 2.5, the unanimous verdict was: revert and split.

Failure 1: The Overloaded State

The .transcribing state was used both for model loading and for actual speech-to-text. Users saw a spinner during model load when they expected to be recording. Same enum case, two completely different meanings.

Failure 2: The Race Condition

Press PTT → model starts loading (up to 5 seconds) → release PTT → stopRequested = true → model finishes → recording timer reads 0.00s → recording silently discarded. The user never knew it happened.

Failure 3: Wrong Timer During Batch ASR

After PTT release, the recording timer kept running during WhisperKit's 1–5 second batch transcription phase. Users thought they were still recording when they weren't.

Failure 4: Frozen Overlay on Model Load

A patch attempted to transition from "Loading model…" to recording lips when load completed. The overlay's generation counter treated this as a stale operation and discarded it, leaving users staring at a frozen spinner.

Failure 5: Every Fix Broke Something Else

The pipeline mixed state management, audio capture, ASR dispatch, streaming forwarding, VAD monitoring, text processing, and clipboard — all in one 758-line class. Changing model load behavior rippled through everything else.

Lesson 1

Streaming and batch are fundamentally different animals

Parakeet transcribes while you talk. WhisperKit transcribes after you stop. No amount of patching can make one feel like the other.

Lesson 2

Five boolean flags = 32 possible states you can't reason about

stopRequested, startRequestCancelled, isStopping, streamingASRActive, isPreWarmed — each one interacts with every other one.

Lesson 3

External consensus was unanimous

GPT-4o: "You've hit that classic point where adding features to a single pipeline creates more complexity than it's worth." Gemini: "Crafting distinct state machines will minimize the risk of state management bugs."

Lesson 4

"Ship patches now, refactor later" never works

From the buddies session: "The pipeline split is going to be exactly as hard in 2 weeks as it is today, but the patches will have calcified and you'll have more code depending on them."

"Reusing the stopRequested flag is exactly the kind of 'clever' solution that leads to bugs later. Its meaning becomes conditional on the state of the machine when it was set."

— Gemini 2.5 Flash, whisperkit-ux-parity session

03 — TWO-HIGHWAY ARCHITECTURE

One merge point. Everything else is separate.

Both highways start at the same hotkey and end at the same LLM polish step. In between, they are completely independent — different state machines, different transcription logic, different model lifecycles.

Parakeet Lane

Streams audio to FluidAudio during recording. ASR finishes in under 100ms after PTT release. The "Transcribing" phase is invisible to users.

WhisperKit Lane

Accumulates audio samples during recording. Sends everything to WhisperKit's batch decoder after PTT release. Users see a visible "Transcribing…" overlay (1–5 seconds).

Single Merge Point

Both highways converge only at the LLM Polish step. The contract is simple: pass a text string and an optional language code. Everything downstream — store, clipboard, paste — is shared.

04 — SYSTEM BOUNDARIES

What WhisperKit owns vs what's shared

The architecture draws hard lines. WhisperKit gets its own state machine and pipeline class. Everything else stays shared and backend-agnostic.

WhisperKit Owns

WhisperKitPipeline

New class, new file. Conforms to DictationPipeline. Full 8-state machine: idle → loadingModel → ready → recording → transcribing → polishing → complete → error.

WhisperKitPipelineState

Separate enum from Parakeet's PipelineState. Each case has exactly one meaning. .loadingModel and .transcribing are never confused.

Its own LLMPolishStep instance

Configured from the same SettingsManager, but each pipeline holds its own copy. No shared mutable state between pipelines.

Its own SilenceDetector

Silero VAD state persists across chunks — sharing it would cause cross-contamination. WhisperKit creates its own instance.

Shared Infrastructure (No Changes)

Component Why it stays shared
AudioCaptureManager Mic, engine, BT codec switch — all backend-agnostic. Only buffer routing differs.
HotkeyService Sends PipelineEvent to active pipeline. Never knows which backend is active.
RecordingOverlayPanel Reads OverlayIntent enum only. Zero knowledge of pipeline state internals.
PasteService Called after polish completes. Same for both pipelines.
TranscriptStore UUID-keyed, append-only. Thread-safe. Both pipelines write to it.
KeychainManager Read-only for API keys. No per-pipeline state.

The Event-Driven Interface

06 — RISKS AND GUARDRAILS

13 guardrails. All mapped to specific phases.

The Oracle agent compiled these from real failures in the git history. Violating any of them risks repeating the same bugs that caused the original revert.

G1 Never overload state cases — .loadingModel and .transcribing must be distinct, always
G2 No boolean flags as shadow state machines — if you need a flag, you're missing a state
G3 Each backend gets its own pipeline class — no shared TranscriptionPipeline with branching logic
G4 Shared infrastructure stays backend-agnostic — if it grows if whisperKit branches, something is wrong
G5 Overlay communicates via OverlayIntent, never PipelineState — UI never knows which pipeline is running
G6 Model load must not block PTT cleanly — user can cancel during load, gets clean idle state
G7 WhisperKit model path is ~/Documents/huggingface/ — already fixed in WhisperKitSetupService
G8 Hardcode decode defaults — no user-facing temperature/threshold sliders. These were removed once already.
G9 Preserve Parakeet's perfect flow — TranscriptionPipeline unchanged except one conformance line
G10 WhisperKit batch ASR phase must be visibly distinct — show "Transcribing…" for 1–5s, not a frozen app
G11 Build on the two kept commits — model path fix and decode defaults are solid; don't rework them
G12 Tune before you stream — Phase 2 (batch quality) is mandatory before Phase 3 (streaming)
G13 Each pipeline independently testable — removing WhisperKit files must never break Parakeet compilation

Anti-Patterns to Never Repeat

AP1Using the same enum case for two different things — overloaded state enums are a bug factory
AP2Adding boolean flags instead of explicit states — five flags = 32 combinations you can't reason about
AP3Scattering if backend == .whisperKit through shared code — use protocol polymorphism
AP4Patching a streaming-first pipeline to support batch — the foundational assumptions are wrong
AP5Exposing temperature/threshold sliders to users — they don't understand these; optimal values are known
AP7Shipping architectural debt with "refactor it later" — it never happens, and more code gets built on top
07 — KEY DISCOVERY

The language bug hiding in the polish step

ew-9bd: LLMPolishStep ignores the language field

WhisperKit's biggest selling point over Parakeet is multilingual support — English, German, and Tamil. But the shared LLM Polish step, which both highways converge on, currently ignores the language field in TextProcessingContext.

This means a German dictation gets polished with an English-only prompt. The LLM may correct perfectly valid German grammar into something wrong, or produce output that reads awkwardly because the model is applying English conventions to German text.

The fix is scoped to Phase 4 and is mandatory — not optional. The prompt must include language context: "Polish the following German transcript…" For Tamil, a lighter-touch prompt may be needed given weaker LLM coverage of that language.

Where the bug lives

Pipeline/Steps/LLMPolishStep.swift — the process(context:) method receives a TextProcessingContext that includes a language: String? field, but constructs its prompt without using it.

What Parakeet gets away with

Parakeet is English-only. For English dictation, an English-assumed polish prompt works fine. The bug was always there; WhisperKit's multilingual support is the first time it actually matters.

Why it's scoped to Phase 4

The bug doesn't block the core WhisperKit flow — transcription and basic polish still work. Phase 4 (Polish Convergence Hardening) is the right moment to address all merge-point contracts at once.

Why it can't slip further

If WhisperKit ships to German and Tamil users with this bug, every polished transcript in those languages is potentially degraded. The Definition of Done for Phase 4 explicitly requires this fixed.

Language Support: Manual Selection, Not Auto-Detect

EN English

Model: small.en (English-optimized). Default. Best accuracy and speed combination.

DE German

Model: small (multilingual). Auto-switching on language change — may trigger model re-download.

TA Tamil

Model: small (multilingual). Lower-resource language — auto-detect would underperform, so manual selection is required.

Why manual selection? Auto-detect adds latency and is unreliable for short utterances in underrepresented languages. Users know what language they're about to speak.
